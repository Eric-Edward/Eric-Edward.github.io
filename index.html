Docker内容

Docker容器底层隔离

​ docker容器的底层隔离主要通过Linux的namespace和cgroup来进行的，其中通过namespace来进行资源的隔离，使不同的资源只能够看到需要的内容。通过cgroup来约束进程对于资源的使用。详细的可以查看https://www.cnblogs.com/DarkH/p/16366020.html。

Docker的一些知识点

  1. 在pull时关于版本的问题，之前学习docker的时候没有关注这一块的内容。不过这里需要注意，因为一个容器是包含这个软件本身和它运行所需要的环境的。而且一般的运行环境都需要linux，这时如果我们随意的选择Linux版本的话，就可能导致相同功能的两个镜像，大小（size）的差别是很大的。==我们一般推荐在pull镜像的时候，优先选择精简版本的运行环境，例如：alpine，slim==

  2. 注意docker create 命令只是创建一个容器，但是这个容器并不是启动的。

  3. 再写端口映射的时候，xxx：xxx，前面的是宿主机端口，外网可以直接访问。后面的是容器内部的端口

  4. 注意一个容器可以绑定多个宿主机的端口（即写多个-p）

  5. docker run默认是前台启动，所以我们需要手动的添加-d来让容器后台启动（其实docker run -d==docker create+docker start）

  6. 注意之前没有特别的学习过docker logs，主要用于查看后台启动的容器的日志。

  7. 注意我们如果需要使用docker attach这个命令的时候，attach是将当前的一个控制台绑定到容器，当我们控制台与容器断开连接时（即按了ctrl+c），就可能会导致容器停止。

  8. 所以我们一般使用docker exec这个命令来与容器进行交互。在这个命令中我们可以加上本次操作的用户名以及用户组，以及可以加上操作的权限。可以让我们最大程度的对容器进行操作。

  9. 我们在需要修改容器内部文件的时候，可以现在外面配置好，然后使用docker cp的命令来进行文件复制的操作。这样就有利于我们操作的便捷性。

    但是我们需要注意的是，src和dest分别为不同值时，cp命令的效果

    • 当src是一个指定文件时
      • dest如果不存在，则会先创建一个dest的文件，并且dest的内容为src的内容
      • dest不存在且以/结束，报错
      • dest存在且是一个文件，则dest的内容替换为src的内容
      • dest存在且是一个目录，src会复制到dest目录下面
    • 当src是一个目录时
      • dest不存在，则创建一个名为dest的文件夹，并复制src中的内容
      • dest是一个文件时，报错
      • dest是一个目录
        • 如果src路径不是以/.结尾，那么src整个目录会被复制到dest下面
        • 如果src路径是以/.结尾的，那么src中的内容会被复制到dest下面

  10. 我们可以使用docker diff来查看当前的容器内部的内容与之前有什么不同，之后我们可以使用docker commit来提交我们当前的最新版本。具体的指令使用可以参考docker commit –help。

  11. ==需要注意一下，我们之前提到的游离镜像在commit时会出现==，当我们同名的镜像在commit的时候(并且版本号也相同的时候)，之前的镜像的名字和版本都会变成none（即游离镜像），这个时候我们在使用docker image prune即可删除游离镜像。

  12. ==重点：推送镜像==

    1. 注册docker hub并登陆。
    2. 创建一个仓库
    3. 修改本地的容器名为docker hub的用户名/容器名（因为在dockerhub上，每一个镜像都有一个完整的路径，其中官方的路径是docker.io/library/xxx，而用户自己创建的镜像的完整路径是：docker.io/用户名/镜像名:tag。所以，我们在推送自己的镜像之前需要先修改镜像的名称）
    4. 我们在推送之前先需要在控制台上登陆我们的docker，否则无法判断推送到哪个用户下。使用docker login进行登陆。可以看.docker下面的config.json来查看是否登陆成功。
    5. 使用docker push 用户名/镜像名：tag进行推送

  13. 当我们使用u盘等工具进行容器和镜像的传输时，有相应的命令可以使用

    docker export和docker import

    这两个命令是操作容器时使用的，我们可以使用docker export将一个容器中的内容到处为一个tar文件，然后我们可以使用docker import将这个tar文件创建为一个镜像。==但是这个镜像是无法正常启动的，他需要之前容器启动时的命令，我们可以通过docker ps –no-trunc来查看原来的容器的命令==。

    docker save和docker load

    这两个命令是操作镜像用的，我们可以使用docker save直接讲一个镜像设置成一个tar文件，然后使用docker load将这个tar文件直接加载为一个镜像。==注意这个也是需要用之前的命令来启动的==

  14. Docker 容器长久运行的原理。

    我们知道docker容器中跑了一个软件，这个软件会有一个阻塞的进程在一直运行，所以才能够让我们的容器一直运行。如果我们容器中的软件没有一个阻塞的进程的话，容器会启动，完成相应的命令之后就会关闭容器。例如：busybox默认会运行一个sh，在这个sh运行完成之后就会关闭容器。

  15. 我们可以使用docker events来查看docker的事件（包含了几乎所有的内容）

Docker中的一些底层原理

Docker存储

​ docker的镜像和容器的存储是分层，他可以是根据dockerfile的每一句话来生成一层。因为分层之后我们就可以只需要修改当前层的内容即可，底层的内容可以共同进行使用降低存储的成本。

容器与镜像的写时复制机制

​ 我们的容器是在镜像的基础上运行的，一个镜像启动的容器会共同使用这个镜像的底层层次。如果一个容器需要读某个文件的话，直接读取镜像中对应的文件即可。但是如果容器需要对镜像中的某个文件进行修改的话，例如：nginx的配置文件需要修改，这是我们需要在容器中新创建一个文件对应镜像中的文件，当容器需要修改的时候，直接修改自己新建的文件即可。从此之后容器再次访问这个文件时，会直接访问当前容器上层中的文件，而不是镜像中的文件。（==需要注意的是，如果我们要查看的话，需要在宿主机中查看，而不是进入到容器内部来查看。而且无论我们如何修改，底层的镜像是不会改变的==）但是自己并没有实验出来，在外面查看的时候发现进不去/etc/nginx（这里以nginx为例）(最终经过实验后成功了！！！)

Docker如果保存容器中的数据

容器挂载：

​ 每一个容器里面的内容，支持三种挂载的方式：

  1. docker自动在外部创建文件夹并与容器内部的文件夹相关联（Dockerfile VOLUME指令的作用）
  2. 自己手动在外部创建文件夹，并且手动的进行挂载（使用Bind mounts）
  3. 将数据挂载到内存中（tmpfs mounts）

​

​ docker使用 -v进行挂载有一下几种情况（以nginx为例）:

  1. 不挂载：默认访问欢迎页
  2. 使用以/开头的绝对路径进行挂载：会访问绝对路径下面下的index.html
  3. 使用不已/开头的路径进行挂载：会访问默认欢迎页面

​ 下面来解释一下具体的原因，当我们不进行挂载的时候，容器中的数据会默认保存在容器中。当我们的容器删除的时候，容器内的数据也会被相应的删除掉。所以我们必须要进行数据的迁移和挂载。

​ 当我们使用以/开头的绝对路径进行挂载时，容器会将对应路径下面的内容替换为我们设置的绝对路径，注意==这个时候很容器出现空挂载的问题（即容器的配置文件在宿主机中的对应路径下面是一个空的文件夹这样就会导致容器启动失败）==

​ 当我们使用不以/开头的路径进行挂载时，其实就是让docker来帮我们来创建一个以给出的名字命名的卷来存储我们需要启动的容器的数据。==而且，这个卷会复制容器初始的config和一些页面之类的内容==，所以我们用这种方式打开的时候能够看到默认的访问页面。(==注意这里删除了容器之后也不会删除这个卷，其实也是正常的，如果删除了这个卷，那我们创建创建这个卷有什么用呢==)

Dockerfile

FROM指令

​ from指令是用来为我们镜像确定一个基础的环境的语句，而且一个dockerfile中只能够有一个from指令。

ARG指令

​ arg指令是用来设置一些==构建阶段==中使用的临时变量而使用的指令，只有在arg指令设置了之后才能够在之后的构建过程中进行使用。而且我们也可以通过在docker build的时候使用 –build-arg选项来修改我们的arg指令设置的参数。可以在RUN指令进行使用。

ENV指令

​ env指令设置的变量可以在==构建和运行阶段==使用，但是指令的==修改只能够在运行时==通过添加–env来进行修改。

ADD和COPY命令

​ 这两个命令都是从宿主机中选择相应的文件复制到容器中，其中ADD可以从远程remote拉取文件或者从本地复制文件，而copy只能够从本地获取。==需要注意的是，我们在写src的时候，一定要让src在镜像构建时可以访问的上下文范围内，否则将会构建失败。==

​ 这两个命令都可以在复制文件时，添加文件访问的相关权限。==但是需要注意这里添加的用户名和用户组必须是容器或者镜像环境中存在的才可以，否则构建时报错。==

​ 需要注意一点，如果我们的镜像的基础环境中，没有/etc/passwd和/etc/group，并且我们在使用这两个命令时使用用户名和用户组添加相关的权限会出现无法匹配，从而导致报错。但是，如果使用的是数字来设置相关的访问权限的话，就不会导致报错。

WORKDIR指令

​ 我们可以通过这个指令来设置我们之后的一些指令工作的目录。如果不设置这个选项的话，默认情况下为/或者由你的基础环境设置。同时workdir支持相对路径的设置（即可以设置多个，他自己默认讲这些路径结合起来）

CMD和ENTRYPOINT指令

​ 这两个指令都是用于容器启动时的命令，这两个单独作用都是可以的。而且这两个命令都有两种写法（即exec写法和shell写法，其中需要注意的是：==直接使用exec是无法读到ENV设置的参数的，不过我们可以让指令为/bin/sh -c，然后再接上参数就可以读取到相应的参数了==）

​ 其中CMD的优先级是低于ENTRYPOINT的。当这两个都写的时候，==运行的指令为ENTRYPOINT+CMD的内容，且二者合起来的命令是一个正确完整的命令==。

​ 我们可以通过在启动容器时添加相应的参数来修改掉CMD的内容，但是ENTRYPOINT是一定运行的。

​ 当dockerfile中出现多个CMD和ENTRYPOINT的时候，只有最后一个会生效。

Docker网络

​ 在我们的应用中，有时需要两个容器进行通信或者容器可以与外界进行交互。这个时候就会产生一些问题。而我们的docker会先创建一个名为docker0的网络，我们在没有自定义网络的情况下，都会将运行的容器加入到这个网络中。

​ 桥接模式可以看下面这个博客的文章，讲的比较详细：https://www.cnblogs.com/hahaha111122222/p/13370773.html。其中，如果我们使用桥接模式的话，不能够通过域名来进行互通，只能通过ip来进行通信。但是，这样就会有局限性，因为我们每次启动容器时，dhcp给分配的ip可能会不同，这样就可能导致访问的ip出错。

​ 下面我们说一下自定义网络，我们在自定义网络时，可以添加一些子网掩码、网关之类的信息。我们之前使用网络想要实现几个容器之间的通信需要使用–link（其实就是修改/etc/host 来增加一个解析项来使得能够解析到）。但是这种方式会出现一种问题，就是容器重新启动的时候ip可能会发生改变，从而导致这种设置失效。但是如果我们自己创建一个网络的话，每当启动一个容器的时候就将这个容器加入到这个网络中，我们的网络为这个容器分配一个ip。每次重启每次重新分配，这时通过自定义网络的记录，就可以准确的找到需要转发的目的地。==如果我们需要两个网络中的容器进行通信的话，可以使用“docker network connnet 网络名 容器名”的方式来将我们这个容器加入到指定的网络中==

选项式和组合式的区别

选项式：vue2的样式，将数据data、方法methods之类的单独进行书写。

组合式：vue3的样式，数据、方法等内容都写在setup中。

==注意，在setup中无法使用this，而在vue2中是可以使用setup的。==

并且vue2和vue3可以混写，并且能够在data、methods通过this访问到setup中的内容。但是setup是访问不到data、methods中的内容的。

setup的语法糖

比较正式的写法：

<script lang="ts">
  export default{
    name:"Person",
    setup(){
      ......
    }
  }
</script>

在这个地方中，我们需要使用name来确定当前组件的名称，并且在setup定义相关的内容。

在使用语法糖之后就会变得很简单，例如下面展示：

<script lang="ts">
  export default{
    name:"Person"
  }
</script>
<script lang="ts" setup>
  ......
</script>

但是我们可以看到这里使用了两个script来完成这个操作，也不是特别的方便。所以，我们可以加入一个插件来使得script标签中可以直接加入一个name的属性来设置我们组件的名称。

我们需要添加vite-plugin-vue-setup-extend，然后在vite.config.ts中引用，并且在plugins中进行添加。

添加了这个插件之后就可以在我们的文件中可以直接使用name属性。

响应式数据

Ref方法

我们如果需要某个变量是响应式的，就需要使用ref将这个变量包裹起来，并且在template中使用的时候直接使用即可，但是在setup中使用时，需要使用name.value的格式，例如下面：

let name = ref("张三")

function handleName(){
  name.value = "李四"
}

此时如果某个操作调用了handleName的方法的话就会使得name变成李四，并且立即在页面上进行响应。

watch监视数据内容

watch一共可以监视4中类型的数据

  1. ref定义的数据
  2. reactive定义的数据
  3. 函数返回一个值（getter函数）
  4. 一个包含上述内容的数组

但是我们总共能够碰到以下5种情况：

  1. 情况一：监视ref定义的基本类型数据，这个很简单。下面给一个具体的样例：

    <script setup name="Person">
import {ref, watch} from "vue";

let sum = ref(0)

function addSum() {
  sum.value++;
}

let stop = watch(sum, (value, oldValue) => {
  console.log(value, oldValue)
  if (value == 5) stop()
})
</script>

  2. 情况二：监视ref定义的对象类型数据，这个有一个点需要注意。==当我们需要对象内部某个属性值的时候，这时的新值和旧值是相同的，因为此时这里的值是通过对象的地址来进行获取的，只修改对象的属性是不会导致对象的地址发生改变的，所以改属性时，新旧值相同。同理的，当我们整体的去修改一个对象时，此时对象的地址就会发生变化，从而可以看到新旧值是不相同的。==下面给出一个具体样例：

    <template>
  <div>
    <h1>姓名:{{ person.name }}</h1>
    <h1>年龄:{{ person.age }}</h1>
    <button @click="changeName">点我修改名称</button>
    <button @click="changeAge">点我修改年龄</button>
    <button @click="changePerson">点我改头换面</button>
  </div>
</template>
<script setup name="Person">
import {ref, watch} from "vue";

let person = ref({
  name: "张三",
  age: 18
})

function changeName() {
  person.value.name = "李四"
}

function changeAge() {
  person.value.age++
}

function changePerson() {
  person.value = {name: "王五", age: 48}
}

watch(person, (newValue, oldValue) => {
  console.log(newValue, oldValue)
}, {deep: true})

</script>

  3. 情况三：监听reactive创建的对象类型的数据，这个和情况二的区别就是：==reactive创建的对象的监听默认是开启了深层次监听的，而且我们是无法通过option进行关闭的。==

  4. 监视ref或reactive定义的【对象类型】中的某个属性，注意以下内容：

    • 若该属性值是一个基本类型，需要写成函数类型，其实就是地址值。
    • 若该属性值是一个对象类型，那么我们可以直接写，也可以写成函数，但是建议写成函数。

  5. 监视多个数据

    ​ 这里就是将多个内容放在一个数组里面，然后旧值和新值都是以数组的形式呈现。

watchEffect

​ 这个内容和watch的作用很想，但是我们在使用这个时，并不需要指明要监视那些对象，他可以自己去分析需要监视那些属性。

标签的Ref属性

​ 我们可以为每一个html元素或者一个组件添加ref标签，来确定的查看或者操作某个元素。

<template>
  <div>
    <h1 ref="h">姓名</h1>
    <button @click="disPlay">点我显示h1</button>
  </div>
</template>
<script setup name="Person">
import {ref} from "vue";

let h = ref();

function disPlay(){
  console.log(h.value)
}

</script>

​ 不过需要注意的是，==在标签上使用的ref需要和js或者ts中的变量命名是需要相同的。==

当我们为一个组件使用ref时，在父组件中是无法看到子组件中js的内容的，除非子组件通过defineExpose来进行暴露。

TS中的接口和范型

​ 我们在自己定于数据时，很容易用着用着就会写错。所以我们可以使用ts的接口和范型来帮助我们。

ts中的接口和数据库定于一张表很像。定义完成之后将这个接口对外暴露即可。

export interface Person {
    id:string,
    name:string,
    age:number
}

​ 如果我们需要让一个数组或者别的容器中的所有的内容都是这个Person类型的话就需要使用范型。其中范型的使用也有两种方式。

  1. 在我们的代码中直接使用范型例如
let persons:Array<Person> = [...]

这样就可以让persons中的每一个人都是符合我们自定义的Person的

  1. 在我们定义的types/index.ts中进行定义

vue3使用props进行参数传递

​ 这里可以有三种方式进行传递，如下所示：

  1. 只接收一个数据。具体情况如下所示

  2. 接收一个数据并且限制这个数据的类型。

    父组件
<template>
  <People :ren="ren1"/>
</template>
<script setup name="Person" lang="ts">
import {type Person} from "@/types";
import People from "@/components/People.vue";
let ren1: Person = {

  id: "12345667789"
}
</script>

子组件
<template>

  <h1>年龄:{{ ren.age }}</h1>
</template>
<script lang="ts" setup>
import type {Person} from "@/types";
defineProps<{ ren: Person }>()
</script>
<style scoped>

</style>

    我们可以看到，在父组件中传递时需要将ren这个添加上’:’，因为在vue中，加上这个就相当于去进行解析，而不是单纯的字符串。我们在子组件中进行接受的时候，如果要接受的属性必须是某个特定的类型的时候，我们需要给这个属性加上类型的约束（==注意添加约束的时候需要写在范型里面，里面写了之后，外面就不需要再写了==）

  1. 什么是非功能性需求以及非功能性需求有哪些？

    非功能需求是与系统的运行、性能、可靠性等相关的需求，通常不直接涉及系统功能，但对系统的整体质量和性能有着重要影响。以下是一些常见的非功能需求：

    1. 性能需求： 包括系统的响应时间、吞吐量、并发用户数等。例如，系统需要在500毫秒内响应用户请求。
    2. 可用性需求： 描述系统的可靠性和可用性，通常以百分比或时间表达。例如，系统每年至少可用99.9%的时间。
    3. 可维护性和可扩展性需求： 系统的易维护性和可扩展性。例如，系统应该容易进行修改和添加新功能。
    4. 安全性需求： 包括对数据的保护、访问控制、身份验证等方面的需求。例如，用户敏感数据必须加密存储。
    5. 兼容性需求： 描述系统与其他软硬件组件或系统的集成程度。例如，系统需要与特定浏览器兼容。
    6. 可靠性和容错性需求： 描述系统在面对错误或故障时的表现。例如，系统应该能够自动从硬件故障中恢复。
    7. 可测试性需求： 描述系统易于测试的程度，包括单元测试、集成测试和系统测试。例如，系统应该支持自动化测试。
    8. 用户体验需求： 描述系统的用户界面、易用性和用户满意度。例如，系统的界面应该直观、易学易用。
    9. 法规和标准需求： 描述系统需要符合的法规和标准，例如数据保护法规或行业标准。
    10. 性能需求： 包括系统的响应时间、吞吐量、资源利用率等。例如，系统需要在高并发情况下保持稳定的响应时间。
    11. 资源需求： 描述系统对硬件、软件和其他资源的需求。例如，系统需要至少2GB的内存。

    这些非功能需求对于确保系统的整体性能、安全性和用户满意度至关重要。在项目开发的早期阶段就明确定义和考虑这些需求，有助于避免后期的问题和提高系统的质量。

  4. 质量属性、策略

    质量属性是衡量软件系统特定方面质量的特征，它们通常包括性能、可靠性、可维护性、安全性等。为了满足这些质量属性，软件架构师采用不同的设计策略和决策。以下是一些常见的质量属性及相应的设计策略：

    1. 性能（Performance）：
      • 策略： 使用缓存技术、优化数据库查询、采用并发和并行处理、使用负载均衡器、采用异步处理等，以提高系统响应时间和吞吐量。
    2. 可靠性（Reliability）：
      • 策略： 实现冗余和备份机制、使用事务管理、设计容错机制、定期进行系统备份、监控系统状态以及采用健壮的错误处理。
    3. 可维护性（Maintainability）：
      • 策略： 使用模块化和松散耦合的设计、采用设计模式、提供清晰的文档、实施自动化测试、使用版本控制系统、遵循编码标准。
    4. 安全性（Security）：
      • 策略： 实施身份认证和授权、使用加密技术保护数据传输和存储、进行安全审计、定期进行漏洞扫描、更新系统和依赖项。
    5. 可用性（Availability）：
      • 策略： 设计冗余系统组件、实施负载均衡、采用灾难恢复计划、监控系统性能并进行实时故障检测。
    6. 可扩展性（Scalability）：
      • 策略： 使用水平和垂直扩展策略、采用微服务架构、使用云服务提供商的弹性计算资源。
    7. 可移植性（Portability）：
      • 策略： 使用跨平台的技术和框架、避免平台特定的依赖、尽量使用标准化的编程语言和库。
    8. 灵活性（Flexibility）：
      • 策略： 采用可配置的组件、使用插件架构、实施参数化设计、避免硬编码。

  5. 设计模式：

    • oo设计原则

      1. 单一职责原则（Single Responsibility Principle - SRP）：
        • 原则： 一个类应该只有一个引起变化的原因。
        • 优点： 提高类的可维护性，降低变更引起的风险。
        • 缺点： 可能会导致类的数量增加。
        • 适用场景： 当一个类负责的功能过于庞大，难以管理时。
      2. 开闭原则（Open/Closed Principle - OCP）：
        • 原则： 软件实体（类、模块、函数等）应该对扩展开放，对修改关闭。
        • 优点： 提高系统的可扩展性，降低修改带来的风险。
        • 缺点： 需要一定的抽象设计，增加系统的复杂性。
        • 适用场景： 在系统需要频繁变化的地方，通过扩展而不是修改来适应变化。
      3. 里氏替换原则（Liskov Substitution Principle - LSP）：
        • 原则： 子类必须能够替换其基类而不影响程序的正确性。
        • 优点： 提高代码的可维护性和可扩展性。
        • 缺点： 需要对继承关系进行深入理解和设计。
        • 适用场景： 在有继承关系的场景中，确保子类可以完全替代基类。
      4. 依赖倒置原则（Dependency Inversion Principle - DIP）：
        • 原则： 高层模块不应该依赖于低层模块，二者都应该依赖于抽象；抽象不应该依赖于细节，细节应该依赖于抽象。
        • 优点： 降低模块之间的耦合性，提高系统的可维护性和可扩展性。
        • 缺点： 需要引入抽象层次，增加设计的复杂性。
        • 适用场景： 在模块之间存在依赖关系的场景中，通过抽象解耦。
      5. 接口隔离原则（Interface Segregation Principle - ISP）：
        • 原则： 客户端不应该强制依赖于它们不使用的接口。
        • 优点： 提高接口的灵活性和可维护性。
        • 缺点： 可能会导致接口的数量增加。
        • 适用场景： 当一个接口过于庞大，客户端只使用其中一部分功能时。

    • 创建型模式

      • 工厂模式

        工厂模式主要包含以下几种形式：

        1. 简单工厂模式（Simple Factory Pattern）：
          • 在简单工厂模式中，有一个工厂类，根据传递给它的参数，动态决定应该创建哪个产品类的实例。
          • 这种模式违反了开闭原则，因为每次添加新产品都需要修改工厂类。
        2. 工厂方法模式（Factory Method Pattern）：
          • 在工厂方法模式中，定义一个创建对象的接口，但让实现这个接口的类来决定实例化哪个类。
          • 每个产品都有一个对应的工厂类，客户端通过调用工厂接口来创建产品。
          • 这种模式符合开闭原则，因为添加新产品只需要添加新的工厂类。
        3. 抽象工厂模式（Abstract Factory Pattern）：
          • 抽象工厂模式提供一个接口，用于创建相关或依赖对象的家族，而不需要指定具体类。
          • 客户端使用产品的抽象接口，而不是直接使用具体的实现。
          • 这种模式有助于确保一组相关的产品可以一起使用，并支持产品家族的替换。

        优点：

        • 封装了对象的创建过程，降低了系统的耦合性。
        • 提高了代码的可维护性和可扩展性。
        • 符合开闭原则，对扩展开放，对修改关闭。

        缺点：

        • 引入了额外的类，增加了系统的复杂性。
        • 在增加新产品时，可能需要修改工厂类的代码，违反了开闭原则（特别是简单工厂模式）。

        适用场景：

        • 客户端不关心对象的创建过程，只关心对象的接口。
        • 系统中的一组相关对象需要一起使用，并且希望将它们的创建和使用分离。
        • 需要提供一个统一的接口来创建一组相关或相互依赖的对象。

      • 单例模式

        单例模式是一种创建型设计模式，其主要目的是确保类只有一个实例，并提供全局访问点。这对于需要在系统中的多个部分共享某些资源或状态的情况非常有用。

        优点：

        1. 全局唯一性： 保证一个类只有一个实例，确保全局唯一访问点。
        2. 延迟实例化： 仅在需要时创建实例，提高性能。
        3. 节省资源： 由于只有一个实例，节省了系统资源。

        缺点：

        1. 全局状态： 单例模式引入了全局状态，可能对测试造成影响。
        2. 滥用： 如果过度使用单例模式，可能会导致系统中的耦合问题。

        适用场景：

        1. 需要全局唯一性的对象： 例如配置管理、日志管理等。
        2. 资源共享： 在整个系统中共享资源，避免多次创建相同的对象。

      • 建造者模式

        建造者模式是一种创建型设计模式，旨在将一个复杂对象的构建与其表示分离，使得同样的构建过程可以创建不同的表示。

        优点：

        1. 分离构建过程： 允许你改变一个产品的内部表示，同时不改变构建过程。
        2. 更好的控制： 可以更好地控制构建过程，根据需要构建不同的对象。
        3. 简化构建过程： 客户端代码只需指定需要构建的类型和内容，而无需关心具体的构建过程。

        缺点：

        1. 代码量增加： 引入了新的构建者类和指挥者类，可能导致代码量的增加。

        适用场景：

        1. 复杂对象的构建： 当一个对象的构建过程比较复杂，而且构建过程需要独立于其表示时，可以使用建造者模式。
        2. 需要更好的构建控制： 当需要更好地控制构建过程，或者需要构建不同表示的对象时，建造者模式是一个不错的选择。

        注意：

        ==工厂模式关注创建对象的过程，主要解决对象实例化的封装问题。==

        ==建造者模式关注创建复杂对象的过程，主要解决构建过程的复杂性和灵活性问题。==

      • 原型模式

        原型模式是一种创建型设计模式，其主要目的是通过复制现有对象来创建新的对象，而无需直接使用其构造函数。这种模式涉及创建新对象的过程，通过克隆（复制）现有对象而不是通过实例化新对象来完成。

        主要角色和结构：

        1. 原型接口（Prototype Interface）： 定义了克隆方法的接口，通常包含一个 clone 方法。
        2. 具体原型类（Concrete Prototype）： 实现了原型接口的具体类。该类的实例可以通过克隆创建新的实例。
        3. 客户端（Client）： 使用原型模式的客户端，通过克隆现有对象来创建新的对象。

        工作原理：

        1. 客户端通过原型接口请求克隆方法，传递一个现有对象的实例。
        2. 具体原型类实现克隆方法，将自身复制一份，生成一个新的实例。
        3. 客户端接收到克隆的对象，得到一个新的实例，而不需要了解对象是如何创建的。

        优点：

        1. 减少构造过程： 避免了直接使用构造函数创建对象，通过复制现有对象可以更加灵活、高效地创建新对象。
        2. 简化对象创建： 对象的创建过程被封装在原型类中，客户端只需要关心克隆方法的调用。

        适用场景：

        1. 当一个系统需要独立于其组成部分，从而允许对这些部分进行替换时。
        2. 当要实例化的类在运行时刻指定时，例如通过动态装载。
        3. 当一个类的实例只能有几个不同状态组合中的一种时。

    • 结构型模式

      • 适配器模式

        适配器模式是一种结构型设计模式，它允许将一个类的接口转换成客户端期望的另一个接口。适配器允许原本由于接口不匹配而不能一起工作的类可以协同工作。

        主要角色和结构：

        1. 目标接口（Target Interface）： 客户端期望的接口，适配器通过实现这个接口来使得被适配者能够与客户端协同工作。
        2. 适配器（Adapter）： 实现了目标接口，并包含一个对被适配者的引用。适配器通过调用被适配者的接口方法来实现目标接口。
        3. 被适配者（Adaptee）： 需要被适配的类，它的接口与目标接口不匹配。
        4. 客户端（Client）： 通过目标接口与适配器进行交互，而不直接与被适配者交互。

        工作原理：

        1. 客户端通过目标接口与适配器交互。
        2. 适配器包含一个对被适配者的引用。
        3. 当客户端调用目标接口的方法时，适配器通过调用被适配者的接口方法来实现目标接口。

        优点：

        1. 解耦性： 允许客户端与被适配者独立工作，无需知道被适配者的具体实现。
        2. 复用性： 可以复用已有的类，而无需修改其源代码。
        3. 灵活性： 可以动态选择适配器，使得客户端可以适配多个不同的被适配者。

        适用场景：

        1. 当你希望使用一个已经存在的类，而它的接口不符合你的需求。
        2. 当你希望创建一个可复用的类，与一些不相关的类协同工作。

      • 装饰器模式

        装饰器模式是一种结构型设计模式，它允许动态地将责任附加到对象上。装饰器提供了一种灵活的方式，可以在不改变原始对象接口的情况下，对其功能进行扩展或修改。

        主要角色和结构：

        1. 组件（Component）： 定义一个对象接口，可以动态地给对象添加职责。
        2. 具体组件（Concrete Component）： 实现了组件接口的具体对象，是被装饰的原始对象。
        3. 装饰器（Decorator）： 持有一个指向组件对象的引用，并实现与组件接口一致的接口。可以有多个具体装饰器，它们可以按一定的顺序进行组合。
        4. 具体装饰器（Concrete Decorator）： 实现了装饰器接口的具体对象，包含一个指向被装饰对象的引用。具体装饰器通常会在调用父类的方法前后添加额外的功能。

        工作原理：

        1. 客户端通过组件接口与具体组件进行交互。
        2. 装饰器类继承自组件接口，持有一个指向具体组件的引用。
        3. 具体装饰器通过调用被装饰对象的方法，并在调用前后添加额外的功能。

        优点：

        1. 灵活性： 可以动态地添加或修改对象的行为，而无需修改其源代码。
        2. 复用性： 可以通过组合不同的装饰器实现不同的行为组合，提高代码的复用性。
        3. 开闭原则： 可以在不修改现有代码的情况下扩展对象的行为。

        适用场景：

        1. 当你希望动态地给对象添加额外的职责，而不影响其接口。
        2. 当你希望通过组合不同的装饰器，实现不同的行为组合。

      • 代理模式

        代理模式是一种结构型设计模式，它允许通过代理对象控制对另一个对象的访问。这种类型的设计模式属于结构型模式，因为它通过组合的方式创建了一个包含原始对象的代理对象。

        工作原理：

        1. 客户端通过与代理对象接口交互。
        2. 代理对象包含一个指向真实对象的引用。
        3. 代理对象在处理客户端请求时，可以在调用真实对象的方法前后添加额外的逻辑。

        优点：

        1. 控制访问： 代理可以控制对真实对象的访问，例如在访问前后执行一些操作。
        2. 解耦： 代理模式可以将客户端与真实对象解耦，客户端只需与代理对象交互即可。

        缺点：

        1. 增加复杂性： 引入了代理对象，可能会增加系统的复杂性。
        2. 性能损耗： 在一些情况下，代理模式可能引入额外的性能开销。

        适用场景：

        1. 当需要控制对对象的访问，并在访问前后执行一些额外的操作时。
        2. 当需要在不改变现有代码的情况下，引入新的功能或修改原有功能时。

      • 外观模式

        外观模式（Facade Pattern）是一种结构型设计模式，它提供了一个简化系统接口的高层接口，使系统更易于使用。外观模式定义了一个高层接口，这个接口使得子系统更容易使用。

        主要角色和结构：

        1. 外观（Facade）： 提供了一个简化系统操作的高层接口，它包含了对子系统各组件的引用，并负责协调它们的工作。
        2. 子系统（Subsystems）： 实现了系统的各个组件，处理外观对象指派的任务。子系统并不知道外观的存在，对于子系统而言，外观仅仅是另一个客户端而已。

        工作原理：

        1. 客户端通过与外观对象交互来完成任务。
        2. 外观对象将任务分派给子系统的各个组件，协调它们的工作。
        3. 客户端无需直接与子系统的各个组件交互，而是通过外观对象简化了与系统的交互。

        优点：

        1. 简化接口： 外观模式提供了一个简化的高层接口，隐藏了系统的复杂性。
        2. 解耦： 外观模式将客户端与子系统解耦，客户端无需了解子系统的具体实现。
        3. 易于使用： 外观模式使得系统更易于使用，客户端只需与外观对象交互，而无需了解系统内部的复杂逻辑。

        缺点：

        1. 违反开闭原则： 如果需要添加新的功能，可能需要修改外观类。
        2. 不符合单一职责原则： 外观对象通常需要知道子系统的各个组件，可能承担过多的职责。

        适用场景：

        1. 当需要为复杂的子系统提供一个简化的接口时。
        2. 当希望将系统的复杂性与客户端分离，使客户端更容易使用系统时。
        3. 当需要划分层次结构时，使用外观模式定义每个层的入口点。

      • 桥接模式

        桥接模式（Bridge Pattern）是一种结构型设计模式，它将一个抽象部分与其实现部分分离，使它们可以独立地变化。这种模式涉及到一个作为桥梁的接口，使得抽象部分和实现部分可以独立地变化而互不影响。

        优点：

        1. 解耦抽象和实现： 桥接模式将抽象和实现分离，使它们可以独立变化，降低了它们之间的耦合度。
        2. 可扩展性： 可以独立地扩展抽象和实现，不影响彼此。
        3. 隐藏实现细节： 客户端只需关心抽象部分，不需要了解具体的实现细节。

        缺点：

        1. 增加复杂性： 桥接模式增加了系统的复杂性，需要额外定义抽象和实现两个层次。

        适用场景：

        1. 当系统有多个独立变化的维度，且希望独立管理这些维度时。
        2. 当希望在抽象和实现部分之间建立稳定的连接，但又不希望使用继承时。
        3. 当希望在多个层次之间建立联系时，例如 GUI 系统中的窗口和不同操作系统的实现。

        示例： 考虑一个图形绘制的例子，有不同的图形（圆、矩形）和不同的颜色（红色、蓝色），桥接模式可以将图形和颜色分离，通过组合的方式创建不同的图形。这样，即使增加新的图形或颜色，也不会影响彼此。

      • 组合模式

        组合模式（Composite Pattern）是一种结构型设计模式，它允许将对象组合成树形结构来表示“部分-整体”的层次结构。通过组合，客户端可以统一对待单个对象和对象的组合。

        主要角色和结构：

        1. 组件（Component）： 定义组合中所有对象的通用接口，可以是叶子节点或容器节点。
        2. 叶子节点（Leaf）： 表示组合中的叶子节点对象，它没有子节点。
        3. 容器节点（Composite）： 包含叶子节点和/或其他容器节点的对象，可以递归地定义子节点。

        优点：

        1. 统一接口： 客户端对待单个对象和组合对象的方式是一致的，不需要区分。
        2. 灵活性： 可以在运行时动态地添加或移除子节点，灵活地组织和管理对象。
        3. 可扩展性： 可以轻松地添加新的叶子节点或容器节点，扩展系统功能。

        缺点：

        1. 限制类型： 定义统一接口可能会限制组件的类型，因为需要满足组件接口的要求。

        适用场景：

        1. 当希望客户端统一处理单个对象和组合对象时。
        2. 当对象的部分-整体层次结构形成树状结构，并且希望客户端能够一致地处理整体和部分时。
        3. 当希望动态地组织和管理对象时。

        示例： 考虑一个组织结构的例子，其中有部门、员工等不同的对象，可以使用组合模式将它们组织成一个树形结构。这样，可以通过统一的接口处理整个组织，不论是部门还是员工。

      • 享元模式

        享元模式（Flyweight Pattern）是一种结构型设计模式，旨在通过共享尽可能多的相似对象以减少内存和计算开销。它适用于大量细粒度的对象，通过共享这些对象的公共部分来节省内存。

        主要角色和结构：

        1. 享元接口（Flyweight）： 定义了具体享元类需要实现的接口，通过这个接口可以接受并作用于外部状态。
        2. 具体享元（ConcreteFlyweight）： 实现了享元接口，包含内部状态和共享的外部状态。具体享元对象需要是可共享的。
        3. 不可共享的享元（UnsharedConcreteFlyweight）： 通常不被共享的具体享元类，包含内部状态但不包含外部状态。
        4. 享元工厂（FlyweightFactory）： 负责创建和管理享元对象，提供对享元对象的访问。在创建享元对象时，如果存在相同的对象，可以返回已存在的对象，否则创建新对象。

        工作原理：

        1. 内部状态（Intrinsic State）： 存储在享元对象内部且不会随外部环境改变的状态。
        2. 外部状态（Extrinsic State）： 存储在享元对象外部且会随外部环境改变的状态。需要在使用时由客户端传递给享元对象。

        优点：

        1. 节省内存： 通过共享相似对象的内部状态，减少了存储大量相似对象所需的内存。
        2. 提高性能： 减少了相似对象的数量，降低了系统的复杂度，提高了性能。

        缺点：

        1. 增加复杂性： 引入了共享机制，可能会使系统变得更加复杂。
        2. 外部状态管理： 外部状态需要在使用时进行传递，可能需要复杂的管理机制。

        适用场景：

        1. 当一个应用程序使用大量相似对象，占用大量内存时。
        2. 对象的大多数状态可以变为外部状态。
        3. 对象的外部状态相对独立，而且可以在多个对象间共享。

    • 行为型模式

      • 策略模式

        策略模式（Strategy Pattern）是一种行为型设计模式，它定义了一系列算法，将每个算法封装并使它们可以互换，使得算法的变化独立于使用算法的客户端。

        主要角色和结构：

        1. 上下文（Context）： 维护一个对策略对象的引用，并可以切换当前的策略。
        2. 策略接口（Strategy）： 定义了所有支持的算法的公共接口。具体策略类实现了这个接口。
        3. 具体策略（Concrete Strategy）： 实现了策略接口的具体算法。

        工作原理：

        1. 定义一组算法： 将一组算法定义成策略接口。
        2. 封装每个算法： 将每个算法封装成具体策略类。
        3. 维护一个上下文： 上下文类维护一个对策略对象的引用，可以在运行时切换策略。
        4. 客户端使用： 客户端通过上下文使用不同的策略，可以动态选择合适的策略。

        优点：

        1. 算法独立性： 客户端与具体算法解耦，可以独立地改变或替换算法。
        2. 复用性： 可以在不修改现有代码的情况下引入新的算法。
        3. 扩展性： 新的策略可以很容易地添加，不影响已有的策略。

        缺点：

        1. 客户端需要了解策略： 客户端需要了解所有的策略，并选择合适的策略。

        适用场景：

        1. 当一个系统有多个变化的算法，并且需要在运行时动态选择其中一个时。
        2. 当一个类定义中有多种行为，可以使用策略模式来定义这些行为，使得类的对象在运行时可以选择行为。
        3. 当一个系统需要动态地在几种算法中选择一种。

      • 模版方法模式

      • command模式

        Command 模式是一种行为设计模式，它将一个请求封装成一个对象，从而允许使用不同的请求、队列或日志来参数化客户端对象。它也支持可撤销的操作。

        结构：

        • Command（命令）： 声明了执行操作的接口。
        • ConcreteCommand（具体命令）： 实现 Command 接口，具体命令对象，通常会包含接收者对象，并调用接收者的方法来执行具体操作。
        • Client（客户端）： 创建命令对象并设置其接收者。
        • Invoker（调用者）： 要求命令执行请求。
        • Receiver（接收者）： 知道如何实施与执行一个请求相关的操作。

        工作流程：

        1. 客户端创建命令对象： 客户端创建具体的命令对象，并设置该命令对象的接收者。
        2. 命令传递给调用者： 客户端将命令对象传递给调用者。
        3. 调用者执行命令： 调用者执行命令，命令对象调用接收者的方法执行具体操作。

        优点：

        • 解耦发送者和接收者： 命令模式通过将请求发送者和请求接收者解耦，使得发送者不需要知道接收者的具体实现。
        • 支持撤销和重做： 可以轻松实现命令的撤销和重做。

        适用场景：

        • 需要将请求发送者和请求接收者解耦的情况。
        • 需要支持撤销和重做操作的情况。
        • 需要在不同时间指定请求、排队请求、或者对请求进行参数化的情况。

        例子： 假设我们有一个简单的遥控器（调用者），它有几个按钮，每个按钮对应一个命令。每个命令都与一个特定的电器（接收者）关联，比如电灯、音响等。按下按钮后，命令会执行相应的操作，比如打开电灯、关闭音响。这样，我们就可以通过修改命令对象，而不用修改遥控器来增加或修改功能。这就是 Command 模式的应用。

      • 中介者模式

        中介者模式（Mediator Pattern）是一种行为设计模式，它定义了一个对象，该对象封装了一组对象之间的交互方式。中介者模式使对象之间的通信变得松散，从而减少对象之间的直接依赖关系。

        在中介者模式中，对象不再直接相互通信，而是通过中介者进行通信。这可以减少对象之间的耦合，使系统更容易维护和扩展。中介者模式通常用于处理对象之间复杂的通信和控制流程。

        典型的中介者模式包括一个中介者对象，多个同事对象，这些同事对象通过中介者进行通信。当一个同事对象发生变化时，它不直接通知其他同事对象，而是通过中介者进行通知，由中介者来协调处理。

        中介者模式的优点包括降低了对象之间的耦合度，使系统更易于维护和扩展。然而，过度使用中介者模式可能导致中介者对象过于庞大复杂，不利于维护。

        中介者模式的经典实例是图形用户界面（GUI）框架，其中用户界面组件通过中介者进行交互。

      • 责任链模式

        责任链模式（Chain of Responsibility Pattern）是一种行为设计模式，它允许你将请求沿着处理者链进行发送，直至有一个处理者处理该请求。这种模式将接收者对象连接成一条链，每个接收者对象都包含对另一个接收者对象的引用。如果一个对象不能处理请求，它会将请求传递给链上的下一个接收者。

        主要参与对象包括：

        1. 处理者接口（Handler）：定义了处理请求的接口，通常包含一个处理请求的方法（如handleRequest）。
        2. 具体处理者类（ConcreteHandler）：实现了处理者接口，负责处理它所能处理的请求，以及设置下一个处理者的引用。

        责任链模式的特点包括：

        • 请求者并不知道是哪个对象处理了它的请求，处理者的顺序可以动态改变。
        • 可以灵活地添加或修改处理者，不影响已有的处理者和客户端代码。
        • 请求可能因为链上的所有处理者都无法处理而被忽略，也可以被链上的某个处理者成功处理。

        责任链模式通常应用于以下场景：

        • 多个对象可以处理同一请求，但具体哪个对象处理由运行时决定。
        • 在不明确指定接收者的情况下，向多个对象中的一个提交一个请求。
        • 可以动态指定处理请求的对象集合。
