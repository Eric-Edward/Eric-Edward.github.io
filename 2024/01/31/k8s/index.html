    k8s学习过程

    Docker内容

    Docker容器底层隔离

    ​ docker容器的底层隔离主要通过Linux的namespace和cgroup来进行的，其中通过namespace来进行资源的隔离，使不同的资源只能够看到需要的内容。通过cgroup来约束进程对于资源的使用。详细的可以查看https://www.cnblogs.com/DarkH/p/16366020.html。

    Docker的一些知识点

    1. 在pull时关于版本的问题，之前学习docker的时候没有关注这一块的内容。不过这里需要注意，因为一个容器是包含这个软件本身和它运行所需要的环境的。而且一般的运行环境都需要linux，这时如果我们随意的选择Linux版本的话，就可能导致相同功能的两个镜像，大小（size）的差别是很大的。==我们一般推荐在pull镜像的时候，优先选择精简版本的运行环境，例如：alpine，slim==

    2. 注意docker create 命令只是创建一个容器，但是这个容器并不是启动的。

    3. 再写端口映射的时候，xxx：xxx，前面的是宿主机端口，外网可以直接访问。后面的是容器内部的端口

    4. 注意一个容器可以绑定多个宿主机的端口（即写多个-p）

    5. docker run默认是前台启动，所以我们需要手动的添加-d来让容器后台启动（其实docker run -d==docker create+docker start）

    6. 注意之前没有特别的学习过docker logs，主要用于查看后台启动的容器的日志。

    7. 注意我们如果需要使用docker attach这个命令的时候，attach是将当前的一个控制台绑定到容器，当我们控制台与容器断开连接时（即按了ctrl+c），就可能会导致容器停止。

    8. 所以我们一般使用docker exec这个命令来与容器进行交互。在这个命令中我们可以加上本次操作的用户名以及用户组，以及可以加上操作的权限。可以让我们最大程度的对容器进行操作。

    9. 我们在需要修改容器内部文件的时候，可以现在外面配置好，然后使用docker cp的命令来进行文件复制的操作。这样就有利于我们操作的便捷性。

      但是我们需要注意的是，src和dest分别为不同值时，cp命令的效果

      • 当src是一个指定文件时
        • dest如果不存在，则会先创建一个dest的文件，并且dest的内容为src的内容
        • dest不存在且以/结束，报错
        • dest存在且是一个文件，则dest的内容替换为src的内容
        • dest存在且是一个目录，src会复制到dest目录下面
      • 当src是一个目录时
        • dest不存在，则创建一个名为dest的文件夹，并复制src中的内容
        • dest是一个文件时，报错
        • dest是一个目录
          • 如果src路径不是以/.结尾，那么src整个目录会被复制到dest下面
          • 如果src路径是以/.结尾的，那么src中的内容会被复制到dest下面

    10. 我们可以使用docker diff来查看当前的容器内部的内容与之前有什么不同，之后我们可以使用docker commit来提交我们当前的最新版本。具体的指令使用可以参考docker commit –help。

    11. ==需要注意一下，我们之前提到的游离镜像在commit时会出现==，当我们同名的镜像在commit的时候(并且版本号也相同的时候)，之前的镜像的名字和版本都会变成none（即游离镜像），这个时候我们在使用docker image prune即可删除游离镜像。

    12. ==重点：推送镜像==

      1. 注册docker hub并登陆。
      2. 创建一个仓库
      3. 修改本地的容器名为docker hub的用户名/容器名（因为在dockerhub上，每一个镜像都有一个完整的路径，其中官方的路径是docker.io/library/xxx，而用户自己创建的镜像的完整路径是：docker.io/用户名/镜像名:tag。所以，我们在推送自己的镜像之前需要先修改镜像的名称）
      4. 我们在推送之前先需要在控制台上登陆我们的docker，否则无法判断推送到哪个用户下。使用docker login进行登陆。可以看.docker下面的config.json来查看是否登陆成功。
      5. 使用docker push 用户名/镜像名：tag进行推送

    13. 当我们使用u盘等工具进行容器和镜像的传输时，有相应的命令可以使用

      docker export和docker import

      这两个命令是操作容器时使用的，我们可以使用docker export将一个容器中的内容到处为一个tar文件，然后我们可以使用docker import将这个tar文件创建为一个镜像。==但是这个镜像是无法正常启动的，他需要之前容器启动时的命令，我们可以通过docker ps –no-trunc来查看原来的容器的命令==。

      docker save和docker load

      这两个命令是操作镜像用的，我们可以使用docker save直接讲一个镜像设置成一个tar文件，然后使用docker load将这个tar文件直接加载为一个镜像。==注意这个也是需要用之前的命令来启动的==

    14. Docker 容器长久运行的原理。

      我们知道docker容器中跑了一个软件，这个软件会有一个阻塞的进程在一直运行，所以才能够让我们的容器一直运行。如果我们容器中的软件没有一个阻塞的进程的话，容器会启动，完成相应的命令之后就会关闭容器。例如：busybox默认会运行一个sh，在这个sh运行完成之后就会关闭容器。

    15. 我们可以使用docker events来查看docker的事件（包含了几乎所有的内容）

    Docker中的一些底层原理

    Docker存储

    ​ docker的镜像和容器的存储是分层，他可以是根据dockerfile的每一句话来生成一层。因为分层之后我们就可以只需要修改当前层的内容即可，底层的内容可以共同进行使用降低存储的成本。

    容器与镜像的写时复制机制

    ​ 我们的容器是在镜像的基础上运行的，一个镜像启动的容器会共同使用这个镜像的底层层次。如果一个容器需要读某个文件的话，直接读取镜像中对应的文件即可。但是如果容器需要对镜像中的某个文件进行修改的话，例如：nginx的配置文件需要修改，这是我们需要在容器中新创建一个文件对应镜像中的文件，当容器需要修改的时候，直接修改自己新建的文件即可。从此之后容器再次访问这个文件时，会直接访问当前容器上层中的文件，而不是镜像中的文件。（==需要注意的是，如果我们要查看的话，需要在宿主机中查看，而不是进入到容器内部来查看。而且无论我们如何修改，底层的镜像是不会改变的==）但是自己并没有实验出来，在外面查看的时候发现进不去/etc/nginx（这里以nginx为例）(最终经过实验后成功了！！！)

    Docker如果保存容器中的数据

    容器挂载：

    ​ 每一个容器里面的内容，支持三种挂载的方式：

    1. docker自动在外部创建文件夹并与容器内部的文件夹相关联（Dockerfile VOLUME指令的作用）
    2. 自己手动在外部创建文件夹，并且手动的进行挂载（使用Bind mounts）
    3. 将数据挂载到内存中（tmpfs mounts）

    ​

    ​ docker使用 -v进行挂载有一下几种情况（以nginx为例）:

    1. 不挂载：默认访问欢迎页
    2. 使用以/开头的绝对路径进行挂载：会访问绝对路径下面下的index.html
    3. 使用不已/开头的路径进行挂载：会访问默认欢迎页面

    ​ 下面来解释一下具体的原因，当我们不进行挂载的时候，容器中的数据会默认保存在容器中。当我们的容器删除的时候，容器内的数据也会被相应的删除掉。所以我们必须要进行数据的迁移和挂载。

    ​ 当我们使用以/开头的绝对路径进行挂载时，容器会将对应路径下面的内容替换为我们设置的绝对路径，注意==这个时候很容器出现空挂载的问题（即容器的配置文件在宿主机中的对应路径下面是一个空的文件夹这样就会导致容器启动失败）==

    ​ 当我们使用不以/开头的路径进行挂载时，其实就是让docker来帮我们来创建一个以给出的名字命名的卷来存储我们需要启动的容器的数据。==而且，这个卷会复制容器初始的config和一些页面之类的内容==，所以我们用这种方式打开的时候能够看到默认的访问页面。(==注意这里删除了容器之后也不会删除这个卷，其实也是正常的，如果删除了这个卷，那我们创建创建这个卷有什么用呢==)

    Dockerfile

    FROM指令

    ​ from指令是用来为我们镜像确定一个基础的环境的语句，而且一个dockerfile中只能够有一个from指令。

    ARG指令

    ​ arg指令是用来设置一些==构建阶段==中使用的临时变量而使用的指令，只有在arg指令设置了之后才能够在之后的构建过程中进行使用。而且我们也可以通过在docker build的时候使用 –build-arg选项来修改我们的arg指令设置的参数。可以在RUN指令进行使用。

    ENV指令

    ​ env指令设置的变量可以在==构建和运行阶段==使用，但是指令的==修改只能够在运行时==通过添加–env来进行修改。

    ADD和COPY命令

    ​ 这两个命令都是从宿主机中选择相应的文件复制到容器中，其中ADD可以从远程remote拉取文件或者从本地复制文件，而copy只能够从本地获取。==需要注意的是，我们在写src的时候，一定要让src在镜像构建时可以访问的上下文范围内，否则将会构建失败。==

    ​ 这两个命令都可以在复制文件时，添加文件访问的相关权限。==但是需要注意这里添加的用户名和用户组必须是容器或者镜像环境中存在的才可以，否则构建时报错。==

    ​ 需要注意一点，如果我们的镜像的基础环境中，没有/etc/passwd和/etc/group，并且我们在使用这两个命令时使用用户名和用户组添加相关的权限会出现无法匹配，从而导致报错。但是，如果使用的是数字来设置相关的访问权限的话，就不会导致报错。

    WORKDIR指令

    ​ 我们可以通过这个指令来设置我们之后的一些指令工作的目录。如果不设置这个选项的话，默认情况下为/或者由你的基础环境设置。同时workdir支持相对路径的设置（即可以设置多个，他自己默认讲这些路径结合起来）

    CMD和ENTRYPOINT指令

    ​ 这两个指令都是用于容器启动时的命令，这两个单独作用都是可以的。而且这两个命令都有两种写法（即exec写法和shell写法，其中需要注意的是：==直接使用exec是无法读到ENV设置的参数的，不过我们可以让指令为/bin/sh -c，然后再接上参数就可以读取到相应的参数了==）

    ​ 其中CMD的优先级是低于ENTRYPOINT的。当这两个都写的时候，==运行的指令为ENTRYPOINT+CMD的内容，且二者合起来的命令是一个正确完整的命令==。

    ​ 我们可以通过在启动容器时添加相应的参数来修改掉CMD的内容，但是ENTRYPOINT是一定运行的。

    ​ 当dockerfile中出现多个CMD和ENTRYPOINT的时候，只有最后一个会生效。

    Docker网络

    ​ 在我们的应用中，有时需要两个容器进行通信或者容器可以与外界进行交互。这个时候就会产生一些问题。而我们的docker会先创建一个名为docker0的网络，我们在没有自定义网络的情况下，都会将运行的容器加入到这个网络中。

    ​ 桥接模式可以看下面这个博客的文章，讲的比较详细：https://www.cnblogs.com/hahaha111122222/p/13370773.html。其中，如果我们使用桥接模式的话，不能够通过域名来进行互通，只能通过ip来进行通信。但是，这样就会有局限性，因为我们每次启动容器时，dhcp给分配的ip可能会不同，这样就可能导致访问的ip出错。

    ​ 下面我们说一下自定义网络，我们在自定义网络时，可以添加一些子网掩码、网关之类的信息。我们之前使用网络想要实现几个容器之间的通信需要使用–link（其实就是修改/etc/host 来增加一个解析项来使得能够解析到）。但是这种方式会出现一种问题，就是容器重新启动的时候ip可能会发生改变，从而导致这种设置失效。但是如果我们自己创建一个网络的话，每当启动一个容器的时候就将这个容器加入到这个网络中，我们的网络为这个容器分配一个ip。每次重启每次重新分配，这时通过自定义网络的记录，就可以准确的找到需要转发的目的地。==如果我们需要两个网络中的容器进行通信的话，可以使用“docker network connnet 网络名 容器名”的方式来将我们这个容器加入到指定的网络中==
